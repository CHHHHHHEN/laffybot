# Session 切换时 busy 状态残留问题修复计划

## 问题描述

用户在前端页面中选中一个 Session 后，发送信息然后 Session 变为 busy 状态，此时用户点击新建会话，在新的 Session 中，界面的发送按钮仍旧会受到上一个 Session 的 busy 状态影响。

## 问题根因

### 核心问题：流式状态是全局单例，切换会话时未重置

`isStreaming` 是全局状态（非会话级别状态）。当 Session A 进入 busy 状态时，`isStreaming` 被设为 `true`。用户切换到 Session B 时，会话切换逻辑只清空了消息列表，未重置流式状态。

InputBar 组件根据 `isStreaming` 决定显示发送按钮还是取消按钮。因此 `isStreaming=true` 时，新会话也错误地显示取消按钮。

### 次要问题：旧会话 status 缓存未更新

中断 SSE 后，旧会话的 `status` 在 TanStack Query 缓存中仍为 `'busy'`。Session 列表中旧会话仍显示为 busy 状态，用户可能困惑。

## 设计决策

**采用多流架构** — 从单流状态切换到多流状态，每个 Session 独立处理 SSE 流。多个会话可以同时流式输出，用户可以在 Session A 等待回复时切换到 Session B 发新消息。

**切换会话不中断旧流** — 在多流架构下，切换会话时不需要中断旧会话的 SSE 连接，旧会话继续在后台接收数据。

**不应该伪造 idle 状态** — 会话状态来自服务端真实状态，前端缓存应保持与服务端同步。

## 修复方案

### 1. 架构重构：从单流到多流

**状态隔离原则**：所有与会话相关的状态（消息列表、流式状态、连接状态、SSE 控制器、请求标识、内容缓冲）均按会话 ID 隔离，全局仅维护当前活动会话标识。

**当前架构**：全局单例状态，任意会话的流式操作独占全局状态，切换会话时状态残留。

**目标架构**：会话级状态映射，每个会话独立维护自己的流式状态，互不干扰。

**新增状态**：`activeSessionId` 标识当前用户正在查看的会话，用于确定 UI 应展示哪个会话的状态。

### 2. SSE 事件路由

多流架构下，SSE 事件需要路由到正确的会话状态。

**路由策略**：闭包绑定。在建立 SSE 连接时已知会话标识，通过闭包将其绑定到事件回调中，事件回调自动操作对应会话的状态。

**选择理由**：后端仅 `session_start` 事件携带会话标识，其他事件类型不携带。闭包方案在连接建立时即完成路由绑定，无需修改后端协议，也无需依赖事件字段做二次分发。

### 3. 会话切换逻辑

多流架构下，切换会话时的行为：

**不中断旧流**：旧会话的 SSE 连接继续在后台运行，不影响新会话的操作。

**不重置全局状态**：无需重置任何全局状态，因为状态已按会话隔离。

**仅更新活动标识**：更新 `activeSessionId`，UI 组件自动读取对应会话的状态。

**UI 响应**：InputBar 组件根据活动会话的流式状态决定显示发送或取消按钮。

### 4. 消息列表隔离

消息列表按会话隔离存储，不同会话的消息互不干扰。

**组件读取**：MessageList 组件根据 `activeSessionId` 读取对应会话的消息列表。

**历史加载**：历史消息加载结果写入对应会话的消息列表，而非覆盖全局状态。

### 5. 取消操作

取消操作针对当前活动会话，中断该会话的 SSE 流并请求服务端取消。

**操作范围**：仅影响活动会话，不影响其他正在后台运行的会话流。

### 6. 会话删除清理

删除会话时需要清理该会话的所有状态：

**中断连接**：若会话正在流式输出，中断其 SSE 连接。

**清理状态**：从消息映射、流式会话集合、连接状态映射等中移除该会话的数据。

**清理时机**：删除 API 调用成功后执行清理，确保状态与后端一致。

### 7. 错误隔离

多流架构下，一个会话的错误不应影响其他会话：

**连接状态隔离**：每个会话独立维护连接状态，一个会话连接错误不影响其他会话。

**错误提示**：错误提示应标识出错的会话，避免用户混淆。

### 8. 后端状态同步

前端缓存通过 SSE 流事件与服务端状态保持同步。

**同步机制**：SSE 事件（如 `session_start`、`session_end`、`error` 等）携带会话状态变更信息，前端据此更新本地缓存。

**无需轮询**：不依赖定时轮询，状态变更通过 SSE 实时推送。

## 修改范围

**前端**：
- 状态管理层 — 重构为按会话隔离的状态结构，新增活动会话标识，新增会话清理方法
- 页面层 — 适配多流架构，会话切换时更新活动标识而非中断连接
- 输入组件 — 根据当前会话的流式状态显示按钮，取消操作针对活动会话
- 消息列表组件 — 按活动会话读取消息列表

## 设计考量

### AbortController 管理层面

AbortController 需要按会话管理，而非全局单例。

**管理位置**：可在状态管理层内部管理，或在页面层按会话存储。前者便于统一清理，后者保持组件自治。

**生命周期**：AbortController 在发送消息时创建，在流结束、取消或会话删除时销毁。

### 组件接口设计

输入组件需要知道当前会话是否正在流式输出。

**方案一**：组件接收 `isStreaming` 属性，由父组件根据活动会话计算后传入。优点是组件不依赖状态管理层，便于测试和复用。

**方案二**：组件内部根据 `activeSessionId` 从状态管理层读取流式状态。优点是减少属性传递，缺点是增加耦合。

**推荐方案一**：保持组件接口不变，由父组件计算属性值。

### 状态初始化时机

会话的消息列表何时初始化：

**懒加载策略**：首次访问该会话时初始化空数组，历史加载完成后填充。避免预初始化所有会话导致不必要的内存占用。

**访问判断**：通过检查会话是否存在于消息映射中来判断是否需要加载历史。

## 废弃功能清理

架构重构过程中，以下状态和方法将被废弃，需要从代码中移除：

### 废弃状态

| 废弃状态 | 替代方案 | 清理方式 |
|----------|----------|----------|
| 全局 `messages` | 按会话隔离的消息映射 | 直接移除，改为从映射中读取活动会话的消息 |
| 全局 `isStreaming` | 流式会话集合 | 直接移除，改为检查活动会话是否在集合中 |
| 全局 `connectionStatus` | 按会话隔离的连接状态映射 | 直接移除，改为从映射中读取活动会话的状态 |
| 全局 `activeRequestId` | 按会话隔离的请求标识映射 | 直接移除，改为从映射中读取活动会话的请求标识 |
| 模块级缓冲变量 | 按会话隔离的缓冲映射 | 移至状态管理层内部，按会话管理 |

### 废弃方法

| 废弃方法 | 替代方案 | 清理方式 |
|----------|----------|----------|
| `setMessages` | 按会话设置消息 | 改为 `setSessionMessages(sessionId, messages)` |
| `clearMessages` | 按会话清理 | 改为 `cleanupSession(sessionId)` |
| `setIsStreaming` | 流式会话集合操作 | 改为 `startStreaming(sessionId)` / `stopStreaming(sessionId)` |
| `setConnectionStatus` | 按会话设置连接状态 | 改为 `setSessionConnectionStatus(sessionId, status)` |
| `setActiveRequestId` | 按会话设置请求标识 | 改为 `setSessionRequestId(sessionId, requestId)` |
| `appendMessage` | 按会话追加消息 | 改为 `appendSessionMessage(sessionId, message)` |
| `updateLastMessage` | 按会话更新最后消息 | 改为 `updateSessionLastMessage(sessionId, updates)` |

### 废弃组件逻辑

| 废弃逻辑 | 替代方案 | 清理方式 |
|----------|----------|----------|
| 页面层全局 `abortRef` | 按会话管理的 AbortController 映射 | 移至状态管理层或在页面层改为映射存储 |
| 会话切换时清空消息 | 不再清空，切换活动标识即可 | 移除清空逻辑 |
| 会话切换时中断 SSE | 不再中断，旧流继续运行 | 移除中断逻辑 |

### 清理策略

**直接替换**：废弃状态和方法直接被新方案替换，不保留旧接口。理由是这些状态和方法仅在前端内部使用，无外部依赖，直接替换成本最低。

**无向后兼容**：不需要向后兼容旧状态格式，因为状态仅存在于运行时内存，不持久化。

**清理时机**：在实现新架构时同步移除废弃代码，避免新旧代码并存导致混乱。

**测试验证**：清理后需验证所有会话操作流程正常，包括创建、切换、发送消息、取消、删除等。

## 风险评估

| 风险点 | 风险等级 | 缓解措施 |
|--------|----------|----------|
| AbortController 生命周期管理 | 中 | 明确创建与销毁时机 |
| 状态一致性 | 中 | 清理操作在删除成功后执行 |
| 废弃代码遗漏 | 低 | 按废弃清单逐项清理，编译检查未引用 |
