## 概述

心跳机制是 SSE 流式响应的关键组件，负责在连接空闲期间发送心跳事件，防止连接因超时断开，并检测客户端连接状态。

## 架构位置

```
┌─────────────────────────────────────────────────────┐
│                   FastAPI Routes                    │
│                   (SSE Response)                    │
└────────────────────┬────────────────────────────────┘
                     │
                     v
         ┌───────────────────────┐
         │  HeartbeatManager     │  ← 本文档核心
         │  (独立组件)            │
         └───────────┬───────────┘
                     │
                     v
         ┌───────────────────────┐
         │    SSE Connection     │
         └───────────────────────┘
```

**设计决策**：
- **独立组件**：HeartbeatManager 作为独立组件，不依赖 SessionManager 或 AgentRunner
- **职责单一**：仅负责心跳发送和连接状态检测，不处理业务逻辑
- **可配置性**：心跳间隔和超时时间可通过配置调整

> **当前代码注记：** `HeartbeatManager` 已在 `_stream_session_events()` 中接入，使用 `asyncio.wait_for` 在事件间隙检测空闲超时（默认 15s），超时则自动发送 `ping` 事件。

## 核心职责

### 1. 连接保活
在 SSE 连接空闲期间定期发送 `ping` 事件，防止代理服务器或客户端因超时断开连接。

### 2. 连接状态检测
通过事件发送机制间接检测客户端连接状态，写入失败时触发连接关闭流程。

### 3. 空闲检测
监控连接空闲时间，仅在真正空闲时发送心跳，不干扰正常业务事件流。

## 设计原则

### 事件驱动
心跳机制基于事件驱动模型，仅在连接空闲时发送心跳，不干扰正常业务事件流。

### 透明性
心跳事件对客户端透明，客户端应忽略 `ping` 事件，无需特殊处理。

### 资源效率
心跳任务仅在需要时启动，连接关闭后立即清理，避免资源泄漏。

## 心跳事件格式

### ping 事件
```
event: message
data: {"type": "ping", "timestamp": "2024-01-15T10:30:15Z"}
```

**字段说明**：
- `type`: 固定为 `"ping"`
- `timestamp`: 心跳发送时间（ISO 8601 格式）

### 客户端处理建议
客户端应：
- 忽略 `ping` 事件，不触发任何业务逻辑
- 不尝试响应心跳事件
- 仅用于连接保活检测（可选）

## 心跳策略

### 发送时机
当 SSE 连接空闲超过配置的间隔时间（默认 15 秒）时，发送心跳事件。

**空闲定义**：自上一个业务事件（content、tool_call、tool_result 等）发送后，未发送新事件的持续时间。

### 发送频率
- **默认间隔**：15 秒
- **可配置范围**：5-60 秒（通过环境变量配置）
- **调整依据**：根据网络环境和代理服务器超时设置调整

### 与业务事件的协调
- 业务事件优先于心跳事件
- 每次发送业务事件后，重置心跳计时器
- 仅在真正空闲时发送心跳

## 与 SSE 流的协作

### 事件流优先级
业务事件优先于心跳事件：
- 有业务事件时，立即发送，不发送心跳
- 无业务事件时，按间隔发送心跳
- 心跳事件不插入业务事件序列中间

### 事件序列示例
```
event: message
data: {"type": "content", "text": "Hello"}     ← 业务事件

[等待 15 秒]

event: message
data: {"type": "ping", "timestamp": "..."}     ← 心跳事件

[等待 10 秒]

event: message
data: {"type": "content", "text": " world"}    ← 业务事件（中断心跳计时）
```

### 连接生命周期
```
SSE 连接建立 → HeartbeatManager 启动心跳任务 → 业务事件流开始
    ↓
业务事件间隙 → 发送 ping 事件
    ↓
连接断开或超时 → HeartbeatManager 清理资源 → 连接关闭
```

## 配置参数

| 参数 | 默认值 | 环境变量 | 说明 |
|------|--------|----------|------|
| 心跳间隔 | 15 秒 | `LAFFYBOT_HEARTBEAT_INTERVAL_S` | 心跳发送间隔 |
| 最小间隔 | 5 秒 | - | 最小心跳间隔限制 |

**配置方式**：
- 通过环境变量 `LAFFYBOT_HEARTBEAT_INTERVAL_S` 设置心跳间隔
- 如果环境变量未设置或解析失败，使用默认值 15 秒
- 实际值会强制不小于最小间隔 5 秒

## 异常处理

### 连接断开检测
- **检测方式**：写入事件时捕获连接异常（由 FastAPI/SSE 层处理）
- **处理策略**：立即停止心跳任务，清理资源

### 心跳发送失败
- **失败原因**：连接已断开、网络异常
- **处理策略**：停止心跳任务，触发连接关闭流程

### 资源清理
- **清理时机**：调用 `stop()` 方法时立即执行
- **清理方式**：设置停止标志，心跳任务检测到后退出循环

## 与其他组件的关系

### 与 SessionManager 的关系
- **独立运行**：HeartbeatManager 不依赖 SessionManager
- **无状态共享**：心跳机制与会话状态管理完全分离
- **并行工作**：SessionManager 处理业务逻辑，HeartbeatManager 处理连接保活

### 与 AgentRunner 的关系
- **事件源无关**：HeartbeatManager 不关心事件来源（AgentRunner 或其他）
- **透明转发**：AgentRunner 生成的事件流与心跳事件独立

### 与 FastAPI 的关系
- **集成点**：在 SSE 响应生成器中预留心跳逻辑
- **资源管理**：利用 FastAPI 的生命周期管理清理心跳任务
