## 动机

Laffybot 当前为 Web 应用形态，用户需通过浏览器访问。桌面端应用的目标：

1. **原生体验** — 独立窗口、系统托盘、全局快捷键、原生文件对话框
2. **零配置启动** — 双击图标即可使用，无需手动启动浏览器
3. **系统集成** — 与操作系统深度集成（通知、文件关联、开机启动）
4. **分发便利** — 单一安装包，无需用户自行部署

## 技术选型

### 最终方案

| 层面 | 选型 | 版本 |
|------|------|------|
| 桌面框架 | Tauri | v2 |
| 前端 | 复用现有 React SPA | — |
| 后端 | 独立 Python 进程 | — |

### 选型依据

#### Tauri vs Electron

| 维度 | Tauri | Electron |
|------|-------|----------|
| 二进制体积 | ~5MB | ~150MB |
| 内存占用 | 低（系统 WebView） | 高（捆绑 Chromium） |
| 原生能力 | Rust 插件生态 | Node.js 生态 |
| 安全模型 | 严格权限系统 | 相对宽松 |
| 与现有架构兼容 | 完全兼容（零前端改动） | 完全兼容 |

**选择 Tauri 的原因**：
- 体积和内存优势明显
- 项目已有 PWA 支持，Tauri 天然兼容 Vite 构建
- Rust 生态与项目追求高性能的目标一致

#### Tauri v2 vs v1

选择 v2 的原因：
- 新的权限系统（capabilities）更灵活
- 更好的 Linux 支持
- 改进的 IPC 性能

#### Python 后端独立进程 vs Sidecar 嵌入

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| 独立进程 | 用户手动启动后端，Tauri 仅管理前端窗口 | 当前阶段，实现最简单 |
| Sidecar | Tauri 自动启动/管理后端进程 | 未来阶段，更完整的桌面体验 |

**初始阶段选择独立进程模式**，原因：
- 实现复杂度低，不改变现有启动方式
- 便于调试（前后端分离）
- 后续可无缝升级到 Sidecar 模式

## 架构设计

### 整体架构

```
┌─────────────────────────────────────────────────────────┐
│                    Tauri Desktop App                     │
│  ┌───────────────────────────────────────────────────┐  │
│  │              WebView (系统原生)                     │  │
│  │  ┌─────────────────────────────────────────────┐  │  │
│  │  │         React SPA (from ui/dist/)           │  │  │
│  │  │                                              │  │  │
│  │  │  Zustand stores  ←→  fetch API  ──────┐     │  │  │
│  │  │  TanStack Query                        │     │  │  │
│  │  └────────────────────────────────────────│─────┘  │  │
│  └──────────────────────────────────────────│────────┘  │
└─────────────────────────────────────────────│───────────┘
                                              │ HTTP/SSE
                                     ┌────────┴────────┐
                                     │  laffybot server │
                                     │  (uvicorn :8000) │
                                     │                  │
                                     │  FastAPI + SSE   │
                                     │  SessionManager  │
                                     │  ToolRegistry    │
                                     └──────────────────┘
```

**关键设计决策**：
- 前端代码零改动，Tauri 直接加载 `ui/dist/`
- 前后端通信方式不变，继续使用 HTTP + SSE
- WebView 中的 `fetch` API 可正常访问 `localhost:8000`

### 数据流

桌面端与 Web 版数据流完全一致：

```
用户输入 → React 组件
           ↓
     api.ts (fetch POST)
           ↓
     FastAPI 后端处理
           ↓
     SSE 事件流返回
           ↓
     chat-store 更新
           ↓
     React 组件重渲染
```

### 安全模型

Tauri v2 采用权限系统（Capabilities）控制 WebView 能力。

#### 核心权限

| 权限 | 用途 |
|------|------|
| `core:default` | 基础窗口管理 |
| HTTP 请求 | 允许访问 localhost:8000 |

#### Content Security Policy

WebView 需配置 CSP 允许连接后端：
- `connect-src` 需包含 `http://localhost:8000`
- `style-src` 需允许内联样式（Tailwind）
- `script-src` 需允许 `unsafe-eval`（部分库依赖）

#### 跨域处理

后端已配置 `cors_origins: ["*"]`，无需额外修改。Tauri WebView 的 origin 为 `https://tauri.localhost` 或 `tauri://localhost`，均在允许范围内。

## 组件职责

### src-tauri/ 目录结构

```
src-tauri/
├── Cargo.toml          # Rust 依赖声明
├── tauri.conf.json     # Tauri 核心配置（窗口、构建、打包）
├── src/
│   ├── main.rs         # Rust 入口
│   └── lib.rs          # Tauri 应用定义
├── capabilities/
│   └── default.json    # 权限配置
└── icons/              # 多平台图标资源
```

### 配置职责划分

| 配置项 | 位置 | 职责 |
|--------|------|------|
| 窗口尺寸/标题 | `tauri.conf.json` | 桌面端窗口属性 |
| CSP 策略 | `tauri.conf.json` | WebView 安全边界 |
| 权限声明 | `capabilities/*.json` | Tauri API 访问控制 |
| 前端构建 | `vite.config.ts` | 无需修改 |
| 后端端口 | `config.json` | 无需修改 |

## 与 Web 版的差异

| 方面 | Web 版 | 桌面版 |
|------|--------|--------|
| 运行环境 | 浏览器标签页 | 独立窗口 |
| 后端启动 | 用户手动启动 | 同左（阶段一） |
| 离线能力 | PWA 缓存 | 同左 |
| 系统集成 | 无 | 托盘、快捷键（阶段二） |
| 分发方式 | URL 访问 | 安装包 |

## 非目标

以下场景不在当前设计范围内：

- **移动端应用** — PWA 已覆盖移动端需求
- **全离线运行** — 需要本地 LLM 推理，不在当前范围
- **多账户切换** — 当前为单用户模式
- **云端同步** — 数据仅存储在本地 SQLite
